Feature Round: Overview & Approaches

7 features across 4 areas — here's how they connect and the key design decisions

1. Global Default Permission Mode

Add @AppStorage("defaultPermissionMode") to General settings tab. NewSessionDialog reads it as initial selection. Simple, no real alternatives needed.

Approach: Single @AppStorage key, segmented picker in General tab, NewSessionDialog reads on init.
Low complexity — straightforward extension of existing patterns.

2. PR Actions

Approve and comment already exist. Adding: request changes, merge (squash default), mark ready/convert to draft.

A

Action Bar (Recommended)

Horizontal button bar at top of PR detail panel. Contextual buttons based on PR state (e.g. "Mark Ready" only shows for drafts, "Merge" only for approved PRs). Merge button has a dropdown for strategy override.

Pros

  • Always visible, discoverable
  • Contextual — shows only valid actions
  • Familiar GitHub-like pattern

Cons

  • Takes vertical space in detail panel
B

Toolbar Menu

Single "Actions" dropdown button in the PR detail header. All actions in a menu, with destructive actions (merge) separated by divider.

Pros

  • Compact — saves vertical space
  • Clean header

Cons

  • Extra click for every action
  • Less discoverable
C

Hybrid: Primary + Overflow

One or two primary actions as buttons (Approve, Merge) plus a "..." overflow menu for less common actions (request changes, draft toggle). Primary buttons change based on PR state.

Pros

  • Quick access to common actions
  • Clean — not cluttered
  • State-aware primary actions

Cons

  • Slightly more complex logic

3. Auto-Detect PR for Sessions

PRManager.fetchPRForWorktree() already exists and RunwayStore.enrichPRs() calls it. This is about making it more visible and reactive.

Approach: Enhance enrichPRs() to run on session creation and periodically. Show linked PR badge in session row. Clicking the badge opens PR detail drawer. Add "View PR" to session context menu.
Mostly wiring — infrastructure exists.

4. ANSI Color Palette Sync

Themes already define all 16 ANSI colors in TerminalPalette. The question is whether they're actually being applied to SwiftTerm at runtime.

Approach: Verify ANSI palette application in TerminalPane. If not wired, map theme.terminal.ansi[0..15] → SwiftTerm's color settings. Update live when theme changes.
Likely a small fix — verify and wire.

5. Sidebar Context Menus + Drag Reorder

Current: restart/delete on sessions. Adding full context menus for both sessions and projects, plus drag-to-reorder.

Approach: SwiftUI .contextMenu modifiers on session and project rows. Drag reorder via .onMove with a persisted sortOrder: Int column in GRDB for both Session and Project tables. Migration adds the column.
Medium complexity — needs GRDB migration + drag gesture handling in List vs ForEach.

6. Window Title

Already implemented! windowTitle in RunwayApp.swift shows "Runway — {session.title}" when a session is selected.

✓ Already done — no work needed.

7. Responsive Layout with Persisted Widths

Persist sidebar width and PR list/detail split positions across restarts.

D

@AppStorage (Recommended)

Store widths as @AppStorage("sidebarWidth"), @AppStorage("prListWidth"), etc. Read on launch, update on drag. Simple, consistent with existing font/theme storage.

Pros

  • Consistent with existing pattern
  • Simple implementation
  • Instant on launch

Cons

  • NavigationSplitView doesn't expose divider position directly — may need custom split view
E

Custom HSplitView

Replace NavigationSplitView with a custom resizable split. Full control over divider position, persisted via @AppStorage. More work but more control.

Pros

  • Full control over widths
  • Reliable persistence
  • Can animate resizes

Cons

  • Lose NavigationSplitView's built-in sidebar toggle
  • More code to maintain
